# Рецензия на задание 2: Проектирование продукта DevEscrow Baltic

## 1. Общая оценка концепции

### Плюсы
Концепция DevEscrow Baltic решает реальную и острую проблему IT-фриланса в ЕС — отсутствие безопасных сделок для физических лиц без регистрации юрлица. Киллер-фича — комиссия 2% и авто-верификация через GitHub Actions — это сильное УТП на фоне Escrow.com (5-10%) и Useme (медленный вывод). Сегмент 100–2 000€ выбран точно: для мелких сделок существующие решения невыгодны, а для крупных — есть конкуренты.

Автор хорошо проработал Power/Interest и RACI матрицы. Особенно радует наличие арбитража с SLA ≤ 48 часов и учёт регуляторных требований (AML, GDPR) — это демонстрирует понимание юридических рисков Escrow-бизнеса.

### Минусы
Главная слабость — **MVP перегружен**. 5 модулей, 10 User Stories, интеграция с GitHub Actions, Stripe Escrow, SEPA, верификация, staging-превью, арбитраж — для первого запуска это слишком. GitHub Actions и авто-верификация — сложная техническая фича, которая может затянуть релиз на месяцы. Без неё можно запуститься с ручной приёмкой, а автоматизацию добавить позже.

Кроме того, **ничего не сказано про тестирование MVP**: как проверять, что эскроу-система корректно обрабатывает все сценарии (успех, фрод, частичная выплата, отмена)?

### Риски

1. **Юридический (высокий)**: DevEscrow по сути — оператор электронных денег. Для работы с деньгами третьих лиц нужна лицензия EMI (Electronic Money Institution) от Банка Литвы или ЦБ ЕС. Получение лицензии — 6–18 месяцев и тысячи евро. Без неё проект нелегален. В отчёте упомянута лицензия EMI в RACI, но scope MVP её получение не включает.

2. **Технический (средний)**: GitHub Actions как верификатор — рискованная зависимость. Если у заказчика нет репозитория (Telegram-бот, Figma-макет, консультация), авто-верификация бесполезна. А если есть — код может быть в GitLab или Bitbucket. Привязка только к GitHub сужает аудиторию.

3. **Бизнес-риск (средний)**: Прозрачность для заказчиков (Keep Informed) без мобильного приложения и встроенного чата. Вся коммуникация идёт через email + Telegram — это может снизить конверсию, так как пользователи привыкли к интерактивному интерфейсу (как у Toptal или Fiverr).

## 2. Аудит артефактов

### Vision & Scope
Vision Statement составлен качественно по шаблону. Scope чётко разделяет in/out of scope. 

**Пробелы:**
- В Scope MVP не указано, будет ли поддержка английского и литовского языков. Если платформа только на литовском — это ограничивает рынок ЕС.
- «Регистрация сделки без юрлица» есть в scope, но не указано, как бороться с фродом (один фрилансер — 10 аккаунтов).
- **Границы НЕ прописаны**: непонятно, какие суммы сделок обрабатываются сразу, а какие уходят на ручную проверку. Без этого AML-политика нереализуема.

### User Stories и Acceptance Criteria

**US-1.1** (Регистрация без юрлица): 
- ✅ Соответствует INVEST (Independent, Negotiable, Valuable, Estimable, Small, Testable).
- ❌ **Пробел**: AC не описывает, что происходит при вводе невалидного email. Система должна показать ошибку, но не указан формат проверки («отправить письмо» vs «проверить синтаксис до отправки»).

**US-2.1** (Создание сделки):
- ✅ Хорошие AC: сумма 10–50 000€, статус Draft.
- ❌ **Пробел**: AC говорят «статус Draft до принятия второй стороной», но не указано, **что происходит, если вторая сторона не приняла сделку в течение N дней**. Сделка зависает в Draft навсегда? Нужен таймер авто-отмены (например, 14 дней).

**US-4.1** (Авто-верификация GitHub Actions):
- ❌ **Логический пробел**: AC «GitHub Actions запускаются при пуше» — но кто делает пуш? Если фрилансер пушит после дедлайна, система должна как-то это обработать. Также не указано, какие credentials использует система для доступа к репозиторию заказчика. Это серьёзная security-дыра.
- Стоит отметить, что AC перегружены: Should Have для MVP — спорно.

### Диаграммы
Sequence Diagram, BPMN и Flowchart присутствуют в текстовом виде. 

**Что не так:**
- Нотация BPMN не соблюдена: нет泳 пулов (pools) и дорожек (lanes), события не разделены на start/end/intermediate. По сути, это текстовый список шагов, а не BPMN-диаграмма.
- Sequence Diagram не содержит временных меток, условий ошибок (таймаут платежа, отказ Stripe).
- Flowchart дублирует BPMN — это избыточно. Одна качественная диаграмма покрыла бы всё.
- **Тупиковые состояния**: в BPMN на ветке «Заказчик не принял → Спор → Арбитраж» нет явного завершения. Что происходит после решения модератора? Возврат денег? Частичная выплата? Это не показано.

## 3. Конструктивные предложения

### Фича для усиления продукта
**«Калькулятор доверия»** — перед созданием сделки показывать фрилансеру статистику заказчика (сколько сделок провёл, % успешных приёмок, среднее время реакции). Это снизит риск фрода с обеих сторон и добавит элемент геймификации. Реализуется за 1-2 недели, но сильно повышает доверие к платформе.

### Упрощение архитектуры
**Убрать авто-верификацию GitHub Actions из MVP.** Заменить на:
- Ручную приёмку с таймером (уже есть)
- Staging-превью на 24ч (уже есть)

Это сократит время разработки на 30-40%. GitHub Actions добавить пост-релиз, когда появятся реальные пользователи и станет ясно, какой формат верификации им нужен. Для первого запуска достаточно доказать ценность эскроу, а не автоматизации тестов.

### Этичное завершение
В целом, работа выполнена на высоком уровне. Power/Interest матрица проработана детально, RACI закрывает все процессы. User Stories сформулированы по INVEST, Acceptance Criteria в основном testable. Самое сильное место — понимание юридических ограничений Escrow-бизнеса (AML, EMI, GDPR). Самое слабое — diagrams (текстовый формат) и перегруженность MVP. Эти замечания — не критика, а предложение сфокусироваться: лучше сделать меньше, но без дыр в безопасности и юрисдикции.
